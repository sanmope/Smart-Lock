# Smart Lock Manager

Event-driven data pipeline for IoT smart lock fleet management during shipments. Smart locks report status, location, and security events in real time. The system processes this telemetry through Kafka, stores it in PostgreSQL (operational) and Redshift (analytics), and enables monitoring of individual devices and overall shipment health.

## Architecture Evolution

This project started as a synchronous REST API backed by SQLite — the API handled reads and writes directly against the database.

It was refactored to an **event-driven architecture (EDA)** to solve three problems:

1. **Dual-write inconsistency**: writing to the database and an analytics system from the same endpoint risks partial failures where one succeeds and the other doesn't.
2. **Horizontal scaling**: a stateful API that writes directly to the DB is harder to scale — every instance needs write coordination.
3. **Decoupled consumers**: analytics, monitoring, and operational storage have different latency and throughput requirements. Coupling them in a single write path forces compromises.

### How it works now

```
Smart Lock Device
    |
    v
FastAPI (stateless) --> Kafka (4 topics, Avro)
                            |
                            +--> Consumer A --> PostgreSQL (operational)
                            |                       |
                            |                       +--> Superset / Grafana
                            |
                            +--> PySpark Streaming --> S3 --> Redshift (analytics)
```

- **Write endpoints** (`POST`, `PATCH`) produce a message to Kafka and return `202 Accepted` with a UUID. The API never touches the database on writes.
- **Read endpoints** (`GET`) query PostgreSQL, which is populated by a separate Kafka consumer.
- **Consistency model**: eventual (~1-2s delay between write and read availability). Acceptable for IoT telemetry.
- **IDs**: changed from integer autoincrement to UUIDs generated by the API before producing, since the database no longer assigns IDs.

## Features

- **Lock registration** – Smart locks self-register when assigned to a shipment
- **State reporting** – Locks continuously send status updates (location, battery, tamper alerts)
- **Security events** – Tamper detection, unauthorized unlock attempts, location deviations
- **Shipment health** – Aggregates lock statuses to determine overall shipment state (ok/warning/critical)
- **Traffic simulator** – Generates realistic IoT traffic with GPS routes, battery drain, incidents, and time acceleration
- **Star schema analytics** – Dimension and fact tables in Redshift for deep analytics
- **Streaming + batch processing** – PySpark jobs for real-time and scheduled aggregations

## Tech Stack

- **Python 3.13** / **FastAPI** – Stateless REST API (Kafka producer)
- **Apache Kafka** (KRaft) – Event streaming with Avro serialization
- **PostgreSQL 16** – Operational database (via Kafka consumer)
- **PySpark 3.5** – Structured Streaming and batch processing
- **Amazon Redshift** (PostgreSQL locally) – Data warehouse with star schema
- **Docker Compose** – Local development environment
- **Terraform** – AWS infrastructure (RDS, EC2, S3)

## Project Structure

```
smart_lock/
├── main.py                     # FastAPI app (produces to Kafka, reads from PostgreSQL)
├── db_models.py                # SQLAlchemy models (UUID PKs)
├── schemas.py                  # Pydantic schemas
├── database.py                 # PostgreSQL connection
├── kafka/
│   ├── config.py               # Topics, producer/consumer settings
│   ├── producer.py             # KafkaProducerService (singleton, idempotent)
│   ├── consumer.py             # Kafka -> PostgreSQL writer
│   ├── serializers.py          # Avro serialization with fastavro
│   └── schemas/                # Avro schemas (.avsc)
├── spark/
│   ├── config.py               # SparkSession builder
│   ├── streaming/              # Kafka -> Redshift (3 streaming jobs)
│   ├── batch/                  # PostgreSQL -> Redshift (3 batch jobs)
│   └── utils/                  # Redshift writer, schema registry
├── redshift/
│   ├── migrations/             # DDL: dimensions, facts, materialized views
│   └── queries/                # Analytics queries
├── simulator/                  # Traffic generator (API, Kafka, or JSON output)
├── docker/                     # Dockerfiles (api, consumer, spark)
├── terraform/                  # AWS infra (RDS, EC2, S3)
├── docker-compose.yml          # Local dev: Kafka, PostgreSQL, Spark, S3
└── requirements*.txt           # Dependencies per service
```

## Getting Started

### Prerequisites

- Docker and Docker Compose
- Python 3.12+ (for the simulator)

### Local development

```bash
# Copy environment template
cp .env.example .env

# Start all services
docker compose up -d

# Verify API is running
curl http://localhost:8000/docs
```

### AWS deployment

```bash
cd terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your values

terraform init
terraform apply

# Start local services pointing to AWS
docker compose --env-file .env.aws up -d api consumer
```

### Generate traffic

```bash
pip install httpx

# 10 locks, 3 shipments, 30 seconds of traffic
python -m simulator --locks 10 --shipments 3 --output api --duration 30

# High volume to Kafka with 10x time acceleration
python -m simulator --locks 1000 --shipments 50 --output kafka --duration 600 --acceleration 10
```

## Services

| Service | Port | Description |
|---------|------|-------------|
| API (FastAPI) | 8000 | REST API, Kafka producer |
| Kafka | 9092 | Event streaming (KRaft, no Zookeeper) |
| Kafka UI | 8080 | Topic/message inspector |
| PostgreSQL | 5432 | Operational database |
| Redshift (local) | 5439 | Data warehouse (PostgreSQL) |
| Spark Master UI | 8082 | Spark job monitoring |
| LocalStack (S3) | 4566 | S3 emulation for Spark staging |

## API Endpoints

### Write (produce to Kafka, return 202)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/locks` | Create a new smart lock |
| PATCH | `/locks/{lock_id}/status` | Update lock status |
| POST | `/events` | Log a security event |
| POST | `/shipments` | Create a new shipment |
| POST | `/shipments/{shipment_id}/locks/{lock_id}` | Assign a lock |
| PATCH | `/shipments/{shipment_id}/locks/{lock_id}/release` | Release a lock |

### Read (query PostgreSQL)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/locks/{lock_id}` | Get lock details |
| GET | `/events` | List events (optional `?severity=` filter) |
| GET | `/shipments/{shipment_id}` | Get shipment details |
| GET | `/shipments/{shipment_id}/health` | Aggregated health status |
